North Star

zero2codex should feel like an interactive terminal-native course, not a normal video course or static documentation site. The reference succeeds because it is concrete, game-like, and personal. The Codex version should keep those product patterns while making the content, copy, branding, and implementation original.

Important creative boundary

Do not copy zero2claude's lesson text, exact visual assets, exact source code, or private implementation. Pattern-match the product mechanics and quality bar: progression, interaction, onboarding, gamification, community loop, and beginner-friendly teaching. Write original Codex-specific content.

Learner Promise

Start with no terminal knowledge. Finish with a real deployed project, a working mental model for software, and a repeatable way to collaborate with Codex safely.

Product Promise

Free core course, account-backed progress, browser labs, review mode, achievements, learner map, community wishes, and a blog that compounds SEO.

Business Promise

Use the free course as a trust-building digital asset. Later offers can include templates, workshops, cohort support, founder build packs, or Codex setup services.

Codex Learning Scope

Official OpenAI Codex docs position Codex as a coding agent that can write, understand, review, debug, and automate software development work. The course should teach Codex as a full development workflow, not just a chat box.

Core Surfaces

  • Codex desktop app for project threads, plans, browser previews, worktrees, diff review, and artifacts.
  • Codex CLI for terminal-first local work, slash commands, permissions, review, resume, and exec automation.
  • IDE extension for editor-attached context and selected-code workflows.
  • Cloud tasks for isolated parallel work after the repo is pushed to GitHub.

Core Habits

  • Prompt with goal, context, constraints, and done criteria.
  • Use Plan mode for fuzzy or high-risk work.
  • Let Codex inspect before editing.
  • Ask Codex to test, verify, and review before accepting work.
  • Keep tasks small enough to review.

Core Safety

  • Understand sandbox modes: read-only, workspace-write, and full access.
  • Understand approval policies and why network access is constrained.
  • Treat web pages, dependencies, and issue text as untrusted.
  • Keep secrets out of prompts and repositories.
  • Use Git to review and roll back changes.

Lesson Engine

The lesson engine is the heart of the product. Build it before you build the dashboard, forum, admin, or AI plan. Every lesson is a sequence of small sections. Each section has a clear interaction, a completion condition, and an event for analytics.

Core section types

  • narrative: short explanation, analogy, key points, tip, optional TTS.
  • quiz: multiple choice, keyboard shortcuts, explanation, confetti on correct.
  • fillInBlank: command or concept answer, alternates, reveal after two tries.
  • match: pair concepts with definitions or commands with results.
  • terminalStep: learner must run a command in the virtual terminal.
  • promptTemplate: learner edits or copies a strong Codex prompt.
  • checklist: setup or verification checklist with durable completion.

Lesson shell controls

  • Progress bar with current section and total sections.
  • Back, close, hint, skip, note, help, report bug, and review mode controls.
  • Keyboard shortcuts: Enter/ArrowRight for continue, ArrowLeft for back, Esc for home.
  • Floating cheat sheet when lessons introduce commands.
  • Completion screen with session stats and next lesson CTA.
  • Review mode for completed lessons that shows answers instead of requiring inputs.

Virtual terminal requirements

The terminal should be a browser-safe simulation for early learning. Later lessons can tell learners when to switch to their real terminal and real Codex installation.

Subsystem Requirements
Virtual file system In-memory tree with directories, files, current directory, home directory, path resolution, JSON import/export, and reset per lesson.
Command parser Tokenize quotes, expand variables, support relative/absolute paths, basic globs, pipes, redirects, append redirects, and helpful errors.
Commands pwd, ls, cd, mkdir, touch, rm, cp, mv, cat, head, tail, echo, grep, rg, wc, export, env, curl, git, help, clear.
Git simulator Support init, status, add, commit, log, diff, checkout, branch, merge, remote, push, pull, and clone. Keep messages beginner-friendly but close to real Git.
Codex simulator For early Codex lessons, simulate codex, codex --help, /status, /plan, /review, /permissions, and codex exec concepts without needing a real install.
Validation Terminal steps validate command, output, file system state, current directory, Git state, environment variables, or curl mock response.

Design System

The reference uses a dark terminal identity, compact layout, monospace headings, low-radius panels, vivid CTA color, and dense course maps. zero2codex should feel related but not copied. Use a Codex-focused identity: calm black, graphite, green, blue, and amber. Avoid duplicating the exact zero2claude accent system.

Layout

  • Course map as first screen, not a marketing landing page.
  • Compact sticky header with brand, language, theme, map, nav, login, sign-up.
  • Level rows with progress bars and expandable lesson lists.
  • Lesson runner with fixed header, scroll body, and sticky bottom CTA.

Visual Language

  • Monospace for brand, headings, labels, and command-like UI.
  • Readable sans for lesson body text.
  • Terminal chrome for labs and blog cards.
  • 8px or smaller radius for most UI. Use 12px only for modal surfaces.

Accessibility

  • Keyboard-first lessons and quizzes.
  • ARIA roles for quiz radio groups and terminal logs.
  • Font size control and high contrast themes.
  • RTL-aware layout and translatable copy from day one.

Gamification and Retention

The course should reward useful learning behavior, not empty points. Focus on completion, return visits, review, hands-on commands, shipping milestones, and community contribution.

Progress mechanics

  • Level progress bars with completed/skipped counts.
  • Current lesson resume card.
  • Section high-water mark so returning to lessons feels natural.
  • Review queue for lessons completed more than 72 hours ago.
  • Test-out challenge at the start of each level.
  • Cheat sheet generated from commands introduced in the level.

Achievement ideas

  • First Command: run your first terminal command.
  • Path Finder: navigate five directories without a hint.
  • Git Saver: complete your first add/commit cycle.
  • Context Builder: write a prompt with goal, context, constraints, and done criteria.
  • Reviewer: use Codex to review a diff and fix one issue.
  • Shipper: deploy the capstone and write handoff docs.

Build Plan

Build in thin slices. The first public version does not need all 151 lessons. It needs the course map, lesson engine, terminal simulator, first 10 to 20 lessons, and a path to keep adding content without rewriting the app.

  • Create app scaffold.
  • Add lint, format, typecheck, and preview scripts.
  • Add sample lesson JSON and validator.
Phase 1

Static course shell

Build the home course map, level cards, lesson list, route skeleton, dark/light theme, font size control, and localStorage progress.

  • No backend yet.
  • Use content files for levels and lessons.
  • Make every route responsive.

Lesson engine

Implement narrative, quiz, fill-in-blank, match, checklist, code example, and prompt template sections. Add keyboard navigation and review mode.

  • Add completion screen and next lesson flow.

Add VFS, command parser, command history, terminal UI, validation, and terminal steps for Levels 2 to 4.

  • Add grep, pipes, redirects, and wc next.
  • Add Git simulator before Git lessons.
Phase 4

Codex labs

Add simulated Codex commands, prompt builders, diff review exercises, AGENTS.md exercises, permission scenarios, worktree diagrams, and browser QA labs.

  • Keep simulations honest. Mark when learners need the real Codex app or CLI.

Accounts and sync

Add register/login, progress API, refresh tokens, achievement API, dashboard, notes, and review queue.

  • Log every admin publish action.

Content Production System

The biggest risk is not code. It is content volume and quality. Build a repeatable lesson writing workflow so 151 lessons can be produced without losing consistency.

Lesson formula

  1. Open with the learner's current confusion.
  2. Explain the concept in one screen.
  3. Show a concrete example.
  4. Make the learner do something.
  5. Ask a small check question.
  6. End with a useful next step.

Voice rules

  • Beginner-safe, direct, and concrete.
  • No condescension.
  • No jargon before it is earned.
  • Use analogies, then return to the real machine.
  • Show commands exactly and explain what changed.

Validation rules

  • Every lesson has at least one active section.
  • Every command has a description and expected outcome.
  • Every Codex claim links to official OpenAI docs.
  • Every terminal lesson can reset cleanly.
  • Every challenge has success and failure feedback.

Quality Bar

zero2codex is a teaching product. Bugs are not just bugs. They become learner confusion. QA should cover code, content, interaction, accessibility, and pedagogy.

Automated checks

  • Typecheck the app.
  • Lint and format.
  • Validate every lesson JSON file.
  • Run unit tests for terminal parser and VFS.
